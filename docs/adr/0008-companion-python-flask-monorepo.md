# Companion is Python + Flask in the Monorepo, Distributed as a Native App

The Companion (the paired-computer program that receives Captures, transcribes them with local Whisper, and serves transcripts back -- see ADR 0003) lives in the **`companion/` subtree of this repository**, is written in **Python with Flask**, and will be distributed to friends as a **native app bundle**, not a Docker image.

**Monorepo, not a separate repo.** The device and the Companion implement one wire contract (`docs/sync-protocol.md`), which calls itself "the shared source of truth for both sides." The dominant lifetime cost of this pair is protocol drift, and a monorepo kills it: the contract doc lives once, and a wire-format change lands on both sides in a single atomic commit -- no two-PR ordering dance, no version skew. The usual reason to split repos (separate teams or ownership, independent release cadence) does not apply to a solo dev building a contract-coupled pair. The two runtimes coexist fine: `companion/` is a top-level sibling to `main/` and `components/`, and ESP-IDF does not try to build a Python subdir.

**Python, because transcription drives the stack.** The one part of the Companion with real weight is running Whisper; the HTTP glue is trivial in any language. Python has the best local-Whisper story (`faster-whisper`/CTranslate2, mature and documented) with the least friction, so the hard part picks the language and it picks Python.

**Flask over the stdlib `http.server` and over FastAPI.** The skeleton is not throwaway -- it *is* the real Companion (faster-whisper plugs into the same seam), and the `venv`/deps cost is paid the moment Whisper lands regardless. That removes both reasons to hand-roll on `http.server` (which would mean manual routing, `Content-Length` streaming, and JSON/status boilerplate that only grows warts). FastAPI is rejected because the workload has **zero concurrency** -- one device issuing sequential requests over one keep-alive connection -- so its async advantage is wasted, while it adds ASGI/uvicorn/pydantic machinery and a footgun (a blocking Whisper call stalls the event loop). Flask is sync, thread-per-request, with trivial routing and streaming helpers. Transcription runs in a **background worker** either way (the upload POST returns `{stored,size}` fast, per the contract), so the framework never runs Whisper in-request.

**Native app, not Docker.** Discovery depends on the Companion advertising a per-owner mDNS hostname on the LAN (`josh-memo.local`, see ADR 0003 and `CONTEXT.md`). Docker Desktop on macOS/Windows runs containers in a Linux VM, so a container **cannot advertise mDNS onto the physical LAN** the ESP is on (`--network host` only works on a Linux host). Docker's one win -- a uniform artifact -- is cancelled by its inability to do the LAN service-discovery this design is built on, on the very OSes friends run. A native app bundle (e.g. Briefcase) is a host process that binds the real LAN, so mDNS works. Friends run a mix of OSes, so per-OS bundling is more work than a single image would have been -- accepted, because the alternative does not function.

## Considered Options

- **Separate repository for the Companion**: clean runtime split, but reintroduces protocol drift as the default failure mode and forces cross-repo changes for every wire tweak. Ruled out; no separate ownership to justify it.
- **stdlib `http.server`**: zero deps, but the "throwaway skeleton" and "zero-dep" premises that justified it are both false (it is the real server; Whisper brings deps anyway), leaving only its hand-rolled boilerplate. Ruled out.
- **FastAPI + uvicorn**: modern async with typed endpoints, but the single-client sequential workload has no concurrency to exploit, and async invites a Whisper-blocks-the-event-loop footgun. Reconsider only if a genuine need for async streaming or endpoint validation appears.
- **Docker distribution**: uniform cross-OS artifact, but breaks LAN mDNS on Docker-for-Mac/Windows, which the discovery model requires. Ruled out. (If the discovery model ever moved off LAN mDNS, this could be revisited.)
- **Go/Rust single static binary**: best raw distribution story (one native binary, native mDNS), but throws away the Python/Whisper head start and turns the easy part into a bindings-gluing exercise. Ruled out for now; the native Python bundle is the lighter path.

## Notes

- The transcription engine is a C/C++ core regardless of host language (`faster-whisper`/CTranslate2 now; a `whisper.cpp` subprocess remains an escape hatch for native Metal speed on Apple Silicon later, with Python still orchestrating HTTP + storage).
- Skeleton stage carries **no Whisper**: `POST /captures/<name>.wav` stores the WAV (temp path -> fsync -> rename) and immediately writes a stub `<stem>.txt`, so the device's download phase (`GET /transcripts` diff + fetch) is testable end-to-end. The stub-writer is the exact seam real transcription replaces, becoming an async background job at that point.
- Python interpreter pinned via `companion/.python-version` (3.12, for `faster-whisper`/`ctranslate2` wheel coverage); `companion/.venv/` is gitignored; the shared bearer token moves to a gitignored `.env` when auth lands.
- Packaging is a real-Companion concern and does not block the skeleton: the endpoint logic is identical no matter how it is later shipped.
